## 1. Overview

docpact MCP is a server implementing the [Model Context Protocol](https://modelcontextprotocol.io/) that exposes the **Contrato Protocol v1** — a formal contract system for AI-native codebases — to MCP-compatible AI coding agents.

The server enables agents to:
- **Discover** functions, contracts, and business rules (RNs) via semantic or keyword search
- **Validate** code changes against declared contracts before committing
- **Enforce** business rules by running relevant tests as a gate
- **Manage** business rules lifecycle (create, verify conflicts, explain)
- **Report** on contract coverage and rule implementation status

### Architecture Position

```
Source files with CONTRATO:meta blocks
  → Index (pre-computed JSON at .docpact/index.json)
    → MCP Server (this protocol)
      → AI Agent (via JSON-RPC 2.0 / stdio)
```

The server reads from a pre-computed index (`.docpact/index.json`) generated by `docpact index`. The index contains all parsed contracts, business rules, cross-references, and optional semantic embeddings.

## 2. Transport Layer

### 2.1 stdio Transport

The server communicates over **stdin/stdout** using newline-delimited JSON-RPC 2.0 messages.

- **Input:** One JSON-RPC request per line on stdin
- **Output:** One JSON-RPC response per line on stdout
- **Logging:** All diagnostic output goes to stderr (structured logging)
- **Encoding:** UTF-8 throughout

### 2.2 Message Framing

Each message is a single line of JSON terminated by `\n`. No Content-Length headers, no streaming, no chunking.

```
< {"jsonrpc":"2.0","id":1,"method":"initialize","params":{}}
> {"jsonrpc":"2.0","id":1,"result":{"protocolVersion":"2024-11-05",...}}
```

### 2.3 Notification Handling

Messages without an `id` field are **notifications** per JSON-RPC 2.0 spec. The server acknowledges them silently (no response). This is used for lifecycle signals like `notifications/initialized`.

## 3. Protocol Lifecycle

```
┌─────────┐         ┌─────────────┐
│  Client  │         │  docpact MCP │
└────┬─────┘         └──────┬───────┘
     │   initialize          │
     │──────────────────────>│
     │   serverInfo +        │
     │   capabilities        │
     │<──────────────────────│
     │   initialized (notif) │
     │──────────────────────>│
     │                       │
     │   tools/list          │
     │──────────────────────>│
     │   tools[]             │
     │<──────────────────────│
     │                       │
     │   tools/call          │
     │──────────────────────>│
     │   CallToolResult      │
     │<──────────────────────│
     │   ...                 │
     │   shutdown             │
     │──────────────────────>│
     │   null result          │
     │<──────────────────────│
```

### 3.1 Required Sequence

1. Client sends `initialize` → server responds with capabilities
2. Client sends `notifications/initialized` → server acknowledges silently
3. Client may call `tools/list` and `tools/call` in any order
4. Client sends `shutdown` → server responds and exits

Calling `tools/call` before `initialize` results in an error.

## 4. Message Format

### 4.1 Request

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "obtener_rn",
    "arguments": { "rn_id": "RN-TKT-001" }
  }
}
```

| Field    | Type              | Required | Description                      |
|----------|-------------------|----------|----------------------------------|
| jsonrpc  | `"2.0"`           | Yes      | JSON-RPC version                 |
| id       | number \| string  | Yes*     | Request identifier (*absent for notifications*) |
| method   | string            | Yes      | Method name                      |
| params   | object            | No       | Method parameters                |

### 4.2 Success Response

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": { ... }
}
```

### 4.3 Error Response

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32000,
    "message": "Tool desconocida: unknown_tool"
  }
}
```

### 4.4 CallToolResult Format

All `tools/call` responses wrap their payload in the MCP `CallToolResult` envelope:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "{\"key\": \"value\", ...}"
      }
    ]
  }
}
```

The `text` field contains a JSON-stringified representation of the tool's return value. Tools return structured dictionaries; the `text` field is their JSON serialization.

## 5. Methods

### 5.1 `initialize`

Establishes the session. Loads or generates the project index.

**Request:**
```json
{
  "method": "initialize",
  "params": {
    "projectRoot": "/path/to/project",
    "force": false
  }
}
```

| Parameter    | Type    | Default                 | Description                              |
|--------------|---------|--------------------------|------------------------------------------|
| projectRoot  | string  | `$DOCPACT_PROJECT_ROOT` or `.` | Project root directory          |
| force        | boolean | `false`                  | Force index regeneration                 |

**Project Root Resolution Order:**
1. `params.projectRoot` (explicit argument)
2. `$DOCPACT_PROJECT_ROOT` (environment variable)
3. `.` (current working directory)

**Response:**
```json
{
  "protocolVersion": "2024-11-05",
  "capabilities": {
    "tools": {}
  },
  "serverInfo": {
    "name": "docpact-mcp",
    "version": "2.0.0"
  }
}
```

### 5.2 `notifications/initialized`

Client notification indicating it is ready. No response required.

### 5.3 `tools/list`

Returns the full list of available tools with their JSON Schema input definitions.

**Request:** No parameters.

**Response:**
```json
{
  "tools": [
    {
      "name": "obtener_rn",
      "description": "Obtiene el contexto completo de una RN...",
      "inputSchema": {
        "type": "object",
        "properties": {
          "rn_id": {
            "type": "string",
            "description": "ID de la RN. Ej: 'RN-TKT-001'"
          }
        },
        "required": ["rn_id"]
      }
    }
  ]
}
```

### 5.4 `tools/call`

Invokes a tool by name with arguments.

**Request:**
```json
{
  "method": "tools/call",
  "params": {
    "name": "obtener_rn",
    "arguments": { "rn_id": "RN-TKT-001" }
  }
}
```

**Response:** `CallToolResult` (see §4.4).

### 5.5 `shutdown`

Gracefully terminates the session.

**Request:** No parameters.  
**Response:** `null` result.

#### `buscar_por_intencion` — Search by Intent

Searches for functions by natural language description. Uses semantic search (cosine similarity via FastEmbed) combined with keyword matching when embeddings are available; falls back to keyword-only otherwise. Returns top 5 results ranked by relevance score.

**Category:** Discovery  
**Side Effects:** None (read-only)  
**Performance:** <5ms with pre-computed embeddings; keyword fallback is instantaneous.

**Input Schema:**
```json
{
  "type": "object",
  "properties": {
    "intencion": {
      "type": "string",
      "description": "Descripción de lo que necesitás. Ej: 'crear un ticket', 'validar RUT de cliente', 'generar reporte de facturación'"
    }
  },
  "required": ["intencion"]
}
```

**Output Schema:**
```json
{
  "resultados": [ { "funcion": "crear_ticket", "archivo": "...", "..." : "..." } ],
  "scores": [0.8723, 0.6541, 0.4210, 0.3100, 0.1500],
  "busqueda_tipo": "semantica",
  "count": 5,
  "total_en_indice": 42
}
```

| Field          | Type     | Description                                      |
|----------------|----------|--------------------------------------------------|
| resultados     | array    | Top 5 function objects, sorted by score desc     |
| scores         | number[] | Relevance scores [0, 1], parallel to resultados  |
| busqueda_tipo  | string   | `"semantica"` or `"keyword"`                     |
| count          | integer  | Number of results returned                       |
| total_en_indice| integer  | Total functions in the index                     |

#### `validar_cambio` — Validate Change (ENFORCEMENT)

Validates a code change before commit. This is an **enforcement gate**: it verifies that the diff references valid RNs and then executes the specific tests for those RNs. If any test fails, the change is **INVALID** — the agent must NOT commit until all tests pass.

**Category:** Discovery / Enforcement  
**Side Effects:** Runs pytest subprocess (read-only on source; executes tests)  
**Timeout:** 30s per test file

**Input Schema:**
```json
{
  "type": "object",
  "properties": {
    "archivo": {
      "type": "string",
      "description": "Path del archivo modificado. Ej: 'soporte/views/portal.py'"
    },
    "diff": {
      "type": "string",
      "description": "El diff o las líneas añadidas/eliminadas"
    },
    "ejecutar_tests": {
      "type": "boolean",
      "description": "Si true (default), ejecuta los tests relevantes. Si false, solo verifica existencia de RNs.",
      "default": true
    }
  },
  "required": ["archivo", "diff"]
}
```

**Output Schema (approved):**
```json
{
  "valido": true,
  "errores": [],
  "warnings": [],
  "funcs_en_archivo": 3,
  "tests_a_correr": ["tests/rn/test_rn_TKT001.py"],
  "test_results": [
    { "test": "tests/rn/test_rn_TKT001.py", "status": "PASS" }
  ],
  "resumen": "APROBADO: cambio válido, podés commitar"
}
```

**Output Schema (blocked):**
```json
{
  "valido": false,
  "errores": [
    {
      "tipo": "rn_fake",
      "rn": "RN-FAKE-999",
      "mensaje": "RN 'RN-FAKE-999' no existe en REGISTRO.md",
      "accion": "Quita 'RN-FAKE-999' del CONTRATO o agrégala a docs/reglas-del-negocio/REGISTRO.md"
    },
    {
      "tipo": "test_fallido",
      "test": "tests/rn/test_rn_TKT001.py",
      "mensaje": "Test tests/rn/test_rn_TKT001.py FALLÓ — tu cambio rompió el comportamiento existente",
      "output_limpio": "...",
      "accion": "Lee el output del test y corrige tu código para que pase."
    }
  ],
  "warnings": [
    {
      "tipo": "rn_sin_test",
      "rn": "RN-TKT-001",
      "mensaje": "RN 'RN-TKT-001' no tiene test file",
      "accion": "Crea tests/rn/test_rn_TKT001.py con Hypothesis PBT"
    }
  ],
  "funcs_en_archivo": 3,
  "tests_a_correr": ["tests/rn/test_rn_TKT001.py"],
  "test_results": [
    { "test": "tests/rn/test_rn_TKT001.py", "status": "FAIL", "output": "...", "returncode": 1 }
  ],
  "resumen": "BLOQUEADO: 2 error(es). Acciones:\n  1. Quita 'RN-FAKE-999' del CONTRATO...\n  2. Lee el output del test..."
}
```

**Error Types:**

| `tipo`         | Severity | Description                                      |
|----------------|----------|--------------------------------------------------|
| `rn_fake`      | Error    | Diff references an RN that doesn't exist in REGISTRO.md |
| `rn_sin_test`  | Warning  | RN exists but has no test file                   |
| `test_fallido` | Error    | Test execution returned non-zero exit code       |
| `test_timeout` | Error    | Test exceeded 30s timeout                        |

#### `navegar_referencias` — Navigate Cross-References

Resolves a reference to its related entities. The tool auto-detects the reference type:

| Input Pattern           | Detected Type | Returns                                    |
|-------------------------|---------------|--------------------------------------------|
| Starts with `RN-`       | RN            | All functions implementing that RN         |
| Contains `/` or ends `.py` | File      | All functions in that file + RNs they use  |
| Otherwise               | Function      | Matching functions with full context       |

**Category:** Discovery  
**Side Effects:** None (read-only)

**Input Schema:**
```json
{
  "type": "object",
  "properties": {
    "referencia": {
      "type": "string",
      "description": "RN (ej: 'RN-TKT-001'), archivo (ej: 'soporte/views/portal.py'), o función (ej: 'crear_ticket')"
    }
  },
  "required": ["referencia"]
}
```

**Output Schema (RN reference):**
```json
{
  "existe": true,
  "tipo": "rn",
  "rn": {
    "descripcion": "...",
    "funciones": [...],
    "tiene_test": true,
    "test": "...",
    "en_registro": true
  },
  "funciones_que_la_implementan": [
    { "funcion": "crear_ticket", "archivo": "...", "linea": 42 }
  ]
}
```

**Output Schema (file reference):**
```json
{
  "existe": true,
  "tipo": "archivo",
  "archivo": "soporte/views/portal.py",
  "funciones": [ { "funcion": "crear_ticket", "rn_ids": ["RN-TKT-001"], "..." : "..." } ],
  "rns_usadas": ["RN-TKT-001", "RN-TKT-003"]
}
```

**Output Schema (function reference):**
```json
{
  "existe": true,
  "tipo": "funcion",
  "funciones": [ { "funcion": "crear_ticket", "archivo": "...", "rn_ids": ["RN-TKT-001"] } ]
}
```

**Output Schema (not found):**
```json
{
  "existe": false,
  "tipo": "rn",
  "busqueda": "RN-TKT-999"
}
```

#### `modificar_archivo` — Modify File (Guard)

Validates a proposed change against the file's CONTRATOs **before** applying it. If the change violates declared side effects or business rules, it is **rejected**. The agent MUST call this tool before modifying any file with a CONTRATO.

**Category:** Project Context / Enforcement  
**Side Effects:** None (read-only validation)

**Input Schema:**
```json
{
  "type": "object",
  "properties": {
    "archivo": {
      "type": "string",
      "description": "Path del archivo a modificar. Ej: 'soporte/services/tickets.py'"
    },
    "diff": {
      "type": "string",
      "description": "El diff o código nuevo a aplicar"
    }
  },
  "required": ["archivo", "diff"]
}
```

**Output Schema (allowed):**
```json
{
  "allowed": true,
  "message": "Cambio permitido",
  "violations": []
}
```

**Output Schema (rejected):**
```json
{
  "allowed": false,
  "message": "Cambio rechazado: 2 violación(es) detectada(s)",
  "violations": [
    {
      "funcion": "crear_ticket",
      "tipo": "side_effect",
      "mensaje": "Función 'send_mail' no declarada en side_effects del CONTRATO",
      "sugerencia": "Agrega 'email_send' al campo side_effects del CONTRATO de crear_ticket"
    },
    {
      "funcion": "crear_ticket",
      "tipo": "rn",
      "mensaje": "Cambio en lógica de RN-TKT-001 sin actualización de test",
      "sugerencia": "Actualiza tests/rn/test_rn_TKT001.py para reflejar el cambio"
    }
  ]
}
```

**Violation Types:**

| `tipo`         | Description                                                  |
|----------------|--------------------------------------------------------------|
| `side_effect`  | Diff introduces a function call not declared in `side_effects` |
| `rn`           | Diff modifies logic governed by an RN without test update     |

#### `verificar_conflicto` — Check Rule Conflicts

Analyzes whether a proposed new business rule would conflict with existing rules. Detects three conflict types:

| Conflict Type   | Threshold | Description                                      |
|-----------------|-----------|--------------------------------------------------|
| `duplicado`     | > 0.70    | Near-duplicate of an existing rule               |
| `mismo_concepto`| > 0.40    | Same entity/concept, potentially contradictory   |
| `override`      | —         | Affects a function already governed by another RN |

**MUST** be called before `crear_rn`.

**Category:** Rule Management  
**Side Effects:** None (read-only)

**Input Schema:**
```json
{
  "type": "object",
  "properties": {
    "rn_descripcion": {
      "type": "string",
      "description": "Descripción de la RN que querés crear. Ej: 'Los tickets deben tener prioridad asignada antes de 24 horas'"
    }
  },
  "required": ["rn_descripcion"]
}
```

**Output Schema:**
```json
{
  "tiene_conflictos": true,
  "conflictos": [
    {
      "tipo": "duplicado",
      "rn_id": "RN-TKT-001",
      "descripcion": "Los tickets deben tener prioridad asignada",
      "similitud": 0.82,
      "explicacion": "La RN propuesta es muy similar a RN-TKT-001. Podría ser un duplicado.",
      "accion": "Revisá si es la misma regla. Si lo es, usá RN-TKT-001 en lugar de crear una nueva."
    },
    {
      "tipo": "override",
      "rn_id": "RN-TKT-003",
      "descripcion": "Tickets urgentes deben escalarse",
      "funcion_afectada": "crear_ticket",
      "explicacion": "La RN propuesta afecta la función 'crear_ticket', que ya está regulada por RN-TKT-003.",
```

## 8. Error Codes

### 8.1 JSON-RPC Error Codes

The server uses standard JSON-RPC 2.0 error codes plus MCP-specific extensions:

| Code    | Name                  | When                                        |
|---------|-----------------------|---------------------------------------------|
| -32700  | Parse error           | Invalid JSON in request                     |
| -32600  | Invalid Request       | Missing required JSON-RPC fields            |
| -32601  | Method not found      | Unknown method (not `initialize`, `tools/list`, `tools/call`, `shutdown`) |
| -32602  | Invalid params        | Missing or invalid parameters               |
| -32603  | Internal error        | Unexpected server-side failure              |
| -32000  | Server error          | Tool execution error, index load failure     |

### 8.2 Tool-Level Errors

Tools return errors **inside their result payload** (not as JSON-RPC errors) for operational failures. These are distinguishable by the presence of an `"error"` key in the returned dictionary.

```json
{
  "content": [
    {
      "type": "text",
      "text": "{\"error\": \"Índice no cargado\"}"
    }
  ]
}
```

**Common tool-level errors:**

| Error Message Pattern                      | Cause                          | Recovery                                    |
|---------------------------------------------|--------------------------------|---------------------------------------------|
| `"Índice no cargado"`                       | Server not initialized         | Call `initialize` first                     |
| `"Tool desconocida: <name>"`               | Unknown tool name              | Check `tools/list`                          |
| `"Archivo no encontrado: <path>"`          | File doesn't exist             | Verify path                                 |
| `"Formato de ID inválido: '<id>'"`         | RN ID doesn't match `^RN-[\w-]+$` | Use `RN-[CAT]-[NUM]` format            |
| `"La RN '<id>' ya existe."`               | Duplicate RN creation          | Use existing RN                             |
| `"No se encontró función <name> con docstring"` | Function without docstring | Add docstring first                         |
| `"Índice no encontrado"`                   | `.docpact/index.json` missing  | Run `setup_docpact`                         |
| `"Tests excedieron timeout de 60s"`        | Test suite too slow            | Optimize tests                              |
| `"No se pudo generar el briefing"`         | Briefing generation failed     | Check project structure                     |
| `"Error ejecutando verificación: <msg>"`   | Verification crash             | Check source files for syntax errors        |
| `"Error ejecutando tests: <msg>"`          | Test runner crash              | Check test infrastructure                   |
| `"Error generando reporte: <msg>"`         | Report generation failed       | Check index integrity                       |

## 9. Discovery Mechanism

### 9.1 Tool Discovery

Clients discover available tools via the `tools/list` method (§5.3). The response contains all 18 tools with:
- **name:** Unique tool identifier
- **description:** Human-readable purpose and usage guidance
- **inputSchema:** JSON Schema defining the tool's input parameters

The `inputSchema` follows JSON Schema Draft 7 conventions:
- `type: "object"` with `properties` and `required`
- Nested types: `string`, `boolean`, `array` with `items`
- `default` values documented in description (not in schema keyword)

### 9.2 Index Discovery

The server discovers project state from `.docpact/index.json`, which contains:

```json
{
  "funciones": { "<func_key>": { "funcion": "...", "archivo": "...", "linea": 42, "rn_ids": ["..."], "contrato": {...}, "tests": [...] } },
  "rns": { "<rn_id>": { "descripcion": "...", "funciones": [...], "tiene_test": true, "test": "...", "en_registro": true } },
  "busqueda": { "keywords": ["..."] },
  "embeddings": { "funciones": { "<key>": [...] }, "rns": { "<id>": [...] } },
  "stats": { "total_rns": 15, "funciones_con_rn": 12, "rns_con_test": 8 },
  "project_root": "/path/to/project",
  "generated_at": "2026-06-13T12:00:00Z"
}
```

### 9.3 Semantic Search Capability

Semantic search is **optional**. When FastEmbed (`fastembed` with `jina-embeddings-v2-base-es`) is installed, the server:
1. Generates embeddings during index creation
2. Stores them in `index.json` under the `embeddings` key
3. Uses cosine similarity (0.7 weight) combined with keyword matching (0.3 weight)

Without FastEmbed, the server falls back to keyword-only search. Tools report which mode they used via `"busqueda_tipo": "semantica" | "keyword"`.

## 10. Versioning

### 10.1 Protocol Version

The server advertises `protocolVersion: "2024-11-05"` in the `initialize` response, conforming to the MCP specification's protocol version.

### 10.2 Server Version

The server advertises `serverInfo.version: "2.0.0"` in the `initialize` response.

**Semantic versioning applies:**

| Change Type                     | Version Bump | Example                          |
|---------------------------------|--------------|----------------------------------|
| New tool added                  | MINOR        | 2.0.0 → 2.1.0                   |
| Tool input schema changed       | MINOR        | New optional parameter           |
| Tool output schema changed      | MINOR        | New fields in response           |
| Tool removed                    | MAJOR        | 2.x.x → 3.0.0                   |
| Required input changed          | MAJOR        | New required parameter           |
| Error code changed              | PATCH        | 2.0.0 → 2.0.1                   |
| Bug fix (no schema change)      | PATCH        | 2.0.0 → 2.0.1                   |

### 10.3 Index Versioning

The index at `.docpact/index.json` is tied to the docpact package version. Index regeneration is required when upgrading docpact across minor versions.

### 10.4 Contract Protocol Version

The Contrato Protocol itself is **v1** (stable). The MCP server (v2.0.0) is the second major implementation of the protocol transport layer.

## 11. Capabilities

### 11.1 Server Capabilities

```json
{
  "capabilities": {
    "tools": {}
  }
}
```

The server declares support for the `tools` capability. It does **not** declare:
- `prompts` — no prompt templates
- `resources` — no exposed resources (all data accessible via tools)
- `logging` — logging is via stderr, not the MCP logging protocol
- `sampling` — no LLM sampling

### 11.2 Client Requirements

To use this server, clients MUST:
1. Support JSON-RPC 2.0 over stdio transport
2. Call `initialize` before any other method
3. Send `notifications/initialized` after receiving the initialize response
4. Parse `CallToolResult` content envelopes

## 12. Security Considerations

### 12.1 File System Access

The server operates on the local filesystem within the configured `project_root`. It:
- **Reads:** Source files, `.docpact/index.json`, `REGISTRO.md`, test files
- **Writes:** `.docpact/index.json` (index generation), `REGISTRO.md` (via `crear_rn`), `docpact.toml` and directories (via `setup_docpact`)
- **Executes:** `pytest` subprocess (via `validar_cambio` and `ejecutar_tests`)

### 12.2 Environment Variables

| Variable                | Purpose                  | Default |
|-------------------------|--------------------------|---------|
| `DOCPACT_PROJECT_ROOT`  | Project root directory   | `.`     |
| `PYTHONDONTWRITEBYTECODE` | Set during test runs   | —       |

### 12.3 Isolation

The server runs as a single-process stdio daemon. There is no network exposure, no authentication, and no multi-tenant isolation. The trust boundary is the host process.

### 12.4 Input Validation

- RN IDs are validated against `^RN-[\w-]+$`
- File paths are resolved relative to `project_root`
- JSON-RPC messages are parsed with strict JSON decoding
- Unknown methods return error code `-32000`

## 13. Conformance

### 13.1 Server Conformance

A conformant docpact MCP server MUST:
- Implement all 18 tools as specified in §7
- Respond to `initialize`, `tools/list`, `tools/call`, and `shutdown` as specified in §5
- Return `CallToolResult` envelopes for all tool calls
- Use JSON-RPC 2.0 error codes as specified in §8
- Advertise `serverInfo.version` using semantic versioning

### 13.2 Client Conformance

A conformant MCP client MUST:
- Follow the lifecycle sequence in §3
- Handle `CallToolResult` content envelopes
- Treat tool-level errors (presence of `"error"` key in result) as non-fatal

### 13.3 Behavioral Guarantees

| Guarantee                        | Scope           |
|----------------------------------|-----------------|
| Read-only tools never modify state | `obtener_*`, `buscar_*`, `navegar_*`, `listar_*`, `verificar_*`, `explicar_*`, `corregir_*`, `ejecutar_verificacion`, `generar_reporte` |
| `validar_cambio` runs tests as gate | Always, unless `ejecutar_tests: false` |
| `crear_rn` validates ID format   | Always          |
| `crear_rn` does not check conflicts | Caller MUST use `verificar_conflicto` first |
| `modificar_archivo` is advisory  | Returns validation; does not apply the diff |
| `setup_docpact` is idempotent    | Skips existing resources |
| Index regeneration is deterministic | Same input → same output (excluding timestamps) |
